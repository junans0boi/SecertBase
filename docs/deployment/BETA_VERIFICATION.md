# 비공개 베타 검증 및 운영 승격 프로토콜

이 문서는 **실제 커플 3~5쌍 대상 7일 비공개 베타** 진행 방법 및 **검증 완료 후 운영 빌드 승격 절차**를 안내한다.

**최신 업데이트: 2026-07-15 (이슈 #17)**

---

## 1. 7일 비공개 베타 검증 프로토콜

### 1.1 베타 테스터 참여 조건 및 준비
- 대상: 실제 커플 3~5쌍
- 환경: 개인 모바일 기기 (iPhone Safari, Android Chrome 등)
- 데이터베이스: 격리된 테스터 데이터베이스 `secretbase_test`
- 접속 주소: `https://test.secertbase.kro.kr`

### 1.2 핵심 일일 여정 점검 체크리스트
베타 커플이 7일 연속으로 다음 핵심 기능을 수행하며 예외 케이스를 점검한다.

- [ ] **1일차: 가입 및 매칭** — 이메일/Google 회원가입 및 상대방 회원코드 매칭, 대기방 상태 확인
- [ ] **2일차: 프로필 및 D-Day** — 프로필 이모지 수정, 이름/닉네임 변경, 시작일 입력 및 디데이 동기화
- [ ] **3일차: 모먼트 기록** — 사진/비디오 미디어 업로드 및 텍스트 캡션 기록, 태그 작성
- [ ] **4일차: 추억 지도** — 특정 위치 검색, 핀 꼽기, 모먼트 연결, 커플 편집 정합성 확인
- [ ] **5일차: 커플 미니게임** — 윷놀이, 가위바위보, 폭탄돌리기 완주 및 네트워크 끊김(새로고침) 후 재접속 복원 테스트
- [ ] **6일차: 연결 해제 및 재결합** — 애인 연결 해제 후 개인 보관함(히스토리) 확인, 동일 커플 재연결 시 데이터 정상 복원 확인
- [ ] **7일차: 데이터 내보내기 및 회원 탈퇴** — ZIP 파일 히스토리 내보내기 및 미디어 무결성 확인, 최종 회원 탈퇴 시 상대방 강제 로그아웃 및 데이터 Anonymization/Tombstone 검증

## 2. 테스터 백업 및 복구 증적 기록 서식

```text
======================================================================
[베타 검증 백업 및 복구 테스트 증적]
======================================================================
- 검증 일시: YYYY-MM-DD HH:MM:SS
- 검증 담당자: junzzang
- 백업 파일명: backup_YYYYMMDD_HHMMSS.tar.enc
- 복구 대상 DB: secretbase_restore_test
- 복구 대상 업로드 경로: services/realtime-server/uploads_restore_test

[무결성 검증 결과]
- Users 테이블 복구 로우 수: XX개 (합격/불합격)
- Couples 테이블 복구 로우 수: XX개 (합격/불합격)
- map_pins 테이블 복구 로우 수: XX개 (합격/불합격)
- setlog_posts 테이블 복구 로우 수: XX개 (합격/불합격)
- 미디어 파일 개수 일치 여부: 백업본 XX개 / 복구본 XX개 (일치/불일치)

[최종 판정]
- 복구 무결성 검사 성공 여부: SUCCESS / FAIL
======================================================================
```

## 3. 운영 승격 및 배포 절차

베타 테스트에서 발견된 모든 P0/P1 버그가 수정되고, 복구 무결성 검증이 통과하면 아래 단계에 따라 운영 서버로 빌드를 승격한다.

### 3.1 릴리즈 브랜치 병합
운영에 배포할 승인된 커밋을 `main` 브랜치에 병합한다.

```bash
git checkout main
git merge codex/issue-1-migrations-test-seam
git push origin main
```

### 3.2 운영 배포 전 안전 백업 실행
배포 도중 문제 발생 시 롤백을 위해 운영 환경 데이터베이스와 미디어를 최종 백업한다.

```bash
# 운영 DB 백업 실행 (암호화)
export BACKUP_ENCRYPTION_KEY="운영용암호화대칭키"
./scripts/backup.sh
```

### 3.3 운영 서버 배포 실행
운영 서버에 접속하여 배포 스크립트를 수행한다.

```bash
cd ~/SecertBase
# 배포 전 백업 강제 활성화 플래그와 함께 실행
REQUIRE_BACKUP_BEFORE_DEPLOY=true \
BACKUP_ENCRYPTION_KEY="운영용암호화대칭키" \
./scripts/deploy_server.sh
```

### 3.4 배포 후 상태 및 스모크 검사 (Smoke Test)
운영 배포가 완료되면 다음 명령 및 시나리오를 통해 헬스체크를 신속히 수행한다.

1. **상태 API 검증:**
   ```bash
   curl -fsS https://secertbase.kro.kr/health
   ```
2. **PM2 프로세스 정상 구동 확인:**
   ```bash
   pm2 status secretbase-realtime
   ```
3. **Caddy 리버스 프록시 상태 확인:**
   ```bash
   sudo systemctl status caddy
   ```
4. **실제 서비스 접속 스모크 체크:**
   - 모바일 브라우저로 `https://secertbase.kro.kr` 접속 후 로그인 및 매칭 상태가 유지되는지 최종 확인한다.
